# Session Handoff: Gate Zero Framework

## Session Summary

Started as toolkit brainstorm continuation (Categories 2-4) but John reframed the conversation around a more fundamental question: **what does any agent or human need to be productive on a project from the start?** This became "Gate Zero" — a set of documents that must exist before any design or build work begins.

Key insight: the same framework applies in both directions — how we build METIS, AND what METIS enforces for clients. "Eat our own cooking."

Also confirmed: SDLC is too heavy for us, but a gate-based hybrid is right. SDLC discipline at design/architecture level, Agile at execution.

---

## DECISIONS MADE THIS SESSION

### Gate Zero Framework — 5 Documents

All 5 must exist before any design, BPMN, DB design, or coding starts.

| # | Document | What it answers |
|---|---|---|
| 1 | Problem Statement | "What is the question?" — John's principle. No data gathering, no analysis, no design until this is clear. |
| 2 | Assumptions & Constraints | Real hard limits — NOT technology choices. Bootstrap budget, finite context windows, AI model changes outside our control, single human in loop. Tech stack belongs elsewhere. |
| 3 | Stakeholders & Decision Rights | Who decides what, who has final call. Agents need to know escalation path — when to stop and wait vs proceed. Connects to pipelines of work (separate concern). |
| 4 | System Map | C4 Level 1 (Context) + Level 2 (Containers). Already in our design notation stack. Prevents drift, duplication, contradiction. Tells agents what already exists. |
| 5 | Design Principles / Ethos | Rules before anyone builds. For METIS: readable/expandable/maintainable, no eye candy, everything must add value, reporting must be actionable not decorative. |

### Dual-Lens Principle (CONFIRMED)
Gate Zero applies both ways:
- As a checklist for building METIS itself
- As an enforced gate METIS applies to every client engagement

### Honest Gap (IMPORTANT)
All 5 Gate Zero documents are currently incomplete for METIS itself:
- Doc 1: Discussed across sessions but never written cleanly in one place
- Doc 2: Implicit in conversations, not written
- Doc 3: Not written ("John decides everything" is not a document)
- Doc 4: 9 areas exist as a list, no actual system map drawn
- Doc 5: Ethos document started today, not formalised

**This is work that must happen before we go deeper into design.**

### Methodology Decision
Hybrid approach confirmed:
- SDLC discipline at planning and architecture levels
- Agile at execution level (development, testing)
- Stage-gate between phases — deliverables must be validated before proceeding
